Practice with reading code and protocol specs (HTTP, nginx, FTS). **For this semester's particular iteration of the course** we also gained real experience working on a team in a production environment by collaborating with CERN on their FTS and Rucio systems.
\n \n Navigate to all answers for Respondent Id 1713191\n 1713191\n \n 1713191\n I would split up what we learned into two categories: a deep dive into CERNs bespoke network systems through our individual projects, and an overview of the different network layers that Prof. Yang covered in lecture. \r\n\r\nThe CERN project (which were real tickets for their File Transfer Service and their data orchestration platform Rucio) was a great experience that came with the expected challenges of working with production code, which involved debugging Docker containers, getting certs, and reading through a lot of code. Since it was pretty hard to test features that would realistically cover the conditions of a worldwide network before deploying into production, this project forced me to think more deeply about the consequences of design choices I was making instead of just relying on testing. \r\n\r\nIn lecture, we read through source code + architecture of real systems, like NGINX and Kubernetes, which I found really helpful for understanding how to implement some of the higher level concepts we discussed in class (ex. NGINXs finite state machine for processing HTTP headers). We also went in depth into HTTP protocol design and discussed how it handled reliability + authentication (SSL/TLS). The only pset in this course was to build a HTTP server from scratch, which was a lot of fun and tied together many of the concepts we covered in the first half of the course. My favorite topic we learned about was congestion control protocols (AIMD, CUBIC, TCP Reno, BBR) which try to optimize resource allocation while still ensuring fairness and avoiding collapse.

This class is Prof Yangs catch-all class for whatever he feels like teaching in the current semester so thats why reviews across years may seem non-cohesive. This semesters iteration was an experimental one where we worked on small teams of 3-4 on open source tickets for CERNs FTS and Rucio services. We also had one monstrous HTTP PSET (which could be written in any desired systems language e.g. Java, C, C++) we could work on in a partnership for. I will begin by saying that I have so much respect for Yang he is an amazing lecturer, has so much academic and industry knowledge and experience, and really puts a lot of effort into his slides. I also want to acknowledge the incredible amount of effort that Yang put into running such an innovative version of this class when he could have easily just done what he did in previous years. The projects cracked open the hood for me to open-source contribution which was incredibly rewarding and implementing an RFC-compliant HTTP server was an absolute joy of a PSET to work on. However, I will bluntly say that this class was a LOGISTICAL NIGHTMARE that made this semester very frustrating and way more stressful than it needed to be. The projects were moving targets there was so much miscommunication about expectations between us, Yang, and the folks at CERN. Each group had weekly 30-minute meetings with Yang every week, which sounds amazing, but every week I felt like I was getting different advice and constantly changing expectations with information I wish I was told earlier. I would often leave feeling like all my progress in the prior week had been uprooted. Not to mention, Prof Yang also has extremely high standards, often discrediting your existing work, and encouraging us to undertake unrealistic tasks like architectural code changes to an existing production code base or deriving new theoretically-sound algorithms. The HTTP PSET was a whole fiasco in of itself. During the first 6 weeks of classes, we basically had no work and then we were given a monstrous PSET with a deadline 2 weeks later (including Fall Break). Many of us grinded 30+ hours each in partnerships only to find out that this was just a check in and that the final deliverable was due a month and half later. Then, we had to schedule a meeting during reading week where we were expected to run test cases that a lot of students failed because of minor issues that were either not mentioned in the spec or directly contradicted the spec. We were then told to fix our PSETs accordingly in the thick of reading periods and finals. We were also left in the dark about the end-of-semester deliverables for the project until literally the day before the end of finals period. The last-minute expectations added so much stress and threw off my entire finals period plan. Ultimately, Im grateful for what I learned in this class that I probably couldnt have gotten from most other classes but I cannot recommend this class in good faith unless you are okay with the aforementioned logistical pain points. A successful student is one who is ready to step up and take ownership of their work, accept potentially never-ending criticism, and dedicate many countless hours to this class. Unfortunately, that was not me this semester.
